# Bug Fix Workflows

版本：v0.5.1

## 流程目標
將 CI、E2E、線上告警或使用者回報的問題，透過標準化 bug fix 流程快速定位、修復、驗證與追蹤。

## 問題來源
1. CI / E2E 測試失敗
2. Production 監控告警
3. 使用者回報或客服工單
4. 安全掃描或相依套件風險

## 主要角色
- Developer：修正程式問題
- QA：協助重現與驗證
- SRE / Ops：處理環境與部署層問題
- AI Agent：整理上下文並提供修補建議

## 主流程
1. 系統接收失敗事件或外部回報
2. 建立 bug ticket，關聯服務、版本、環境與嚴重度
3. 收集 log、trace、截圖、commit 與最近部署資訊
4. 判斷是否可由 AI Agent 先做根因分析與 patch 建議
5. Developer 實作修正並補上回歸測試
6. 重新執行指定測試與驗證流程
7. 驗證通過後合併並排入部署
8. 關閉 bug，產出修復紀錄與 RCA

## 嚴重度分級
| 等級 | 說明 | SLA 建議 |
|---|---|---|
| P0 | 核心功能中斷或資料損壞 | 立即處理 |
| P1 | 主要功能受影響 | 4 小時內回應 |
| P2 | 一般功能異常 | 1 個工作日內處理 |
| P3 | 低風險優化或邊角問題 | 排入一般排程 |

## 分支流程
### 自動修復候選
1. 測試失敗後觸發 AI bug fix agent
2. 產生 patch、說明與風險摘要
3. 由 Developer 審核後決定是否採納

### Hotfix
1. P0/P1 問題建立 hotfix branch
2. 精簡審核與必要測試後先上修復版本
3. 再回補主分支與完整 RCA

### 環境問題
1. 若問題不在程式碼，轉由 SRE / Ops 處理
2. 保留 incident 記錄與恢復時間
3. 視情況更新監控與告警閾值

## RCA 最低內容
1. 問題時間線
2. 根因描述
3. 修復方式
4. 預防措施
5. 是否需補監控、測試或文件

## 指標建議
1. MTTA
2. MTTR
3. Hotfix 次數
4. 同類問題重複發生率
