# Project Workflows

版本：v0.5.1

## 流程目標
定義從專案建立、規劃、執行、驗收到結案的標準流程，讓 PM、Tech Lead、Developer、QA 與 Stakeholder 在相同節奏下協作。

## 主要角色
- PM：建立專案、維護時程與資源
- Tech Lead：拆解技術範圍與風險
- Developer：執行任務與交付程式
- QA：安排測試與驗收
- Stakeholder：確認需求與最終結果

## 前置條件
1. 已建立使用者、角色與權限
2. 已配置通知通道與專案模板
3. 已具備任務、測試、部署模組可用環境

## 主流程
1. PM 建立專案基本資料，指定負責人、時程、優先級與目標版本
2. PM 與 Stakeholder 確認需求範圍、驗收標準與里程碑
3. Tech Lead 建立模組、版本與技術拆解項目
4. PM/Tech Lead 將工作拆成任務並指派成員
5. Developer 依任務進行開發、提交 commit 與 PR
6. 系統自動觸發 CI、測試與部署流程
7. QA 依版本或里程碑執行驗證與回報問題
8. PM 彙整進度、風險與驗收結果
9. Stakeholder 完成驗收後，專案進入結案或維運階段

## 狀態流轉
| 階段 | 狀態 | 說明 |
|---|---|---|
| 規劃 | Draft | 專案資料建立中 |
| 啟動 | Active | 已開始任務執行 |
| 驗收 | UAT | 功能已完成，等待驗收 |
| 上線 | Released | 已正式發版 |
| 維運 | Maintenance | 進入維護期 |
| 結案 | Closed | 專案已完成 |

## 關鍵分支
### 需求變更
1. PM 建立 change request
2. Tech Lead 評估影響範圍、工時與風險
3. Stakeholder 確認是否納入當前版本
4. 系統更新里程碑、任務與通知

### 時程延遲
1. 專案進度低於門檻時觸發告警
2. PM 重新調整優先級或增補資源
3. 更新預計完成日並通知相關人員

### 驗收失敗
1. QA 或 Stakeholder 回報缺陷
2. 建立 bug 任務並關聯版本
3. 修復後重新執行測試與驗收

## 輸入與輸出
### 輸入
- 專案基本資料
- 需求文件與驗收標準
- 人員配置與里程碑計畫

### 輸出
- 專案看板與進度報告
- 版本與發版記錄
- 驗收結果與結案摘要

## 通知節點
1. 專案建立完成
2. 里程碑到期前提醒
3. 高風險或阻塞告警
4. 驗收完成或退回

## KPI 建議
1. 專案準時率
2. 里程碑達成率
3. 驗收一次通過率
4. 版本延誤次數
