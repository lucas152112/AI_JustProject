# Testing Workflows

版本：v0.5.1

## 流程目標
讓測試從開發提交後自動化接續，並將測試結果、風險與缺陷回饋到任務與版本管理。

## 主要角色
- Developer：修正測試失敗與補足案例
- QA：維護測試計畫與驗收結果
- CI/CD System：自動執行測試流程
- PM：追蹤版本品質狀態

## 測試層級
1. Lint / Static Check
2. Unit Test
3. Integration Test
4. E2E Test
5. UAT / Manual Validation

## 主流程
1. 程式碼 push 或 PR 建立後觸發測試 pipeline
2. 先執行 lint、build 與 unit test
3. 若通過則執行 integration test 與 API 檢查
4. 再進入 E2E 測試、畫面驗證或關鍵路徑驗收
5. 系統彙整報告，更新任務、PR、版本與 Dashboard
6. 若有失敗，建立缺陷或交由 bug fix 流程處理
7. 測試通過後，將版本標記為可部署

## 結果分級
| 等級 | 條件 | 處理方式 |
|---|---|---|
| Passed | 全部關鍵測試通過 | 允許進入部署 |
| Warning | 非關鍵測試失敗 | 需人工判斷是否放行 |
| Failed | 關鍵案例失敗 | 阻擋合併或部署 |
| Blocked | 環境或資料異常 | 重新排程並通知管理者 |

## 失敗處理流程
1. 系統保留 log、trace、畫面截圖與重現資訊
2. 分析失敗屬於程式缺陷、測試腳本問題或環境異常
3. 自動或人工建立 bug 任務
4. 修正完成後重新執行指定測試層級

## 測試報告內容
1. 成功率與失敗率
2. 失敗案例列表
3. 最近趨勢與版本比較
4. 測試耗時與瓶頸分析

## 通知節點
1. 關鍵測試失敗立即通知
2. 測試完成通知 PM、QA、開發者
3. 長時間未完成的 pipeline 告警

## 指標建議
1. 測試通過率
2. 缺陷再開率
3. 平均修復時間
4. 測試環境穩定度
